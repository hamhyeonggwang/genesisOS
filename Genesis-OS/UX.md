## 1. UX 원칙

1. **대화가 곧 설계다.** 핵심 경험은 질문 세션이다. 나머지 화면은 세션의 결과를 보여주고 관리하는 보조 무대다.
2. **결정을 눈에 보이게.** 답변이 확정되는 순간 Decision Panel에 즉시 축적된다. 사용자는 "지금까지 무엇이 정해졌는가"를 항상 본다 — M1→M2 정착의 시각화.
3. **한 화면, 한 질문 (헌법 제5조).** 세션 UI는 동시에 하나의 질문만 활성화한다. 다음 질문은 답변 후에만 나타난다.
4. **백지 금지.** 모든 빈 상태는 다음 행동 버튼과 예시를 제공한다.
5. **모바일 대응, 데스크톱 우선.** 설계 작업의 주 무대는 데스크톱이다. 모바일에서는 열람·답변은 가능하되 문서 편집은 열람 중심으로 단순화한다.

## 2. 질문 세션 UI 결정 (PRD Open Question 1)

**결정: 하이브리드 — 대화 스트림 + 구조화 답변 카드 + Decision Panel.**

| 대안 | 기각/채택 사유 |
|---|---|
| 순수 채팅형 | 기각 — 결정사항이 대화에 파묻혀 휘발 (헌법 제3조 위반 위험) |
| 순수 폼(위저드)형 | 기각 — 답변 반영형 후속 질문(제5조) 불가능, 고정 질문지로 퇴화 |
| **하이브리드 (채택)** | 대화의 유연성 + 폼의 구조화. 본 PRD를 생성한 세션이 골든 레퍼런스 |

## 3. 핵심 플로우

### F1. 새 프로젝트 생성 → Discover 시작
```
대시보드 [+ 새 프로젝트]
 → 아이디어 한 문단 입력 (플레이스홀더에 예시 제공)
 → [시작] 클릭
 → 시스템: projects + 5개 phase 생성, Genesis Memory 질문 은행 로드(V1)
 → Discover 질문 세션으로 자동 진입, AI가 첫 질문 제시
소요 목표: 입력 시작 → 첫 질문까지 30초 이내
```

### F2. 질문 세션 루프 (핵심 플로우)
```
① AI가 질문 카드 제시
   ├ WHY / WHAT / HOW 접이식 설명
   ├ 선택지 2~4개 (각각 설명 포함)
   └ [직접 입력] 자유 텍스트
② PO 답변
③ 시스템: 답변을 결정으로 정착 (context_entries 저장)
   → Decision Panel에 카드 추가 (실시간)
④ AI: 답변을 반영해 다음 질문 생성 → ①로
⑤ 단계 완료 조건 충족 시:
   AI가 "정의 완료" 선언 + 결정 요약 제시
   → [산출물 생성] 버튼 활성화
```

- 질문 건너뛰기 허용 — 단, "미정" 상태로 Decision Panel에 기록 (No Guessing: AI가 임의로 채우지 않음)
- 이전 답변 수정: Decision Panel에서 해당 결정 클릭 → 수정 → 이후 질문에 반영 필요 시 AI가 후속 질문 재생성

### F3. 산출물 생성 → 검토 → 승인
```
[산출물 생성] → 생성 중 화면 (진행 표시, 문서별 순차 생성)
 → 문서 뷰(SC-06)로 이동, phase = in_review
 → PO가 Markdown 직접 편집 또는 "이 부분 다시 생성" 부분 재생성 요청
 → [승인] → phase = done, 다음 단계 잠금 해제
```

### F4. Handoff
```
Engineer 단계 done
 → Handoff 화면: 패키지 구성 미리보기
    (CLAUDE.md · 문서 6종 · 태스크 목록 · 구현 시작 프롬프트)
 → [zip 다운로드] 또는 [구현 프롬프트 복사]
 → 완료 화면: "구현 후 회고를 기록하세요" (Flywheel ③ 예약)
```

### F5. 세션 재개 (컨텍스트 연속성 — User Story 4)
```
재접속 → 대시보드에 "이어서 하기" 카드 (마지막 작업 프로젝트·단계)
 → 클릭 → 세션 진입
 → AI 첫 메시지: 지금까지의 결정 요약 3~5줄 + "다음 질문은 ~였습니다"
 → 곧바로 진행 재개
```

### F6. 회고 기록 (S6)
```
Handoff 후 프로젝트 홈에 [회고 작성] 배지
 → 구조화 폼: 잘된 결정 / 설계 누락(재작업 유발) / 다음에 물어야 할 질문
 → 저장 → retrospectives (V1에서 Genesis Memory 승격 후보가 됨)
```

## 5. 인터랙션 규칙

### 질문 카드
- WHY/WHAT/HOW는 기본 접힘, 제목만 노출 (숙련 사용자 속도 우선). 첫 사용 시(프로젝트 1~2개째)는 기본 펼침
- 선택지에는 항상 설명 1~2문장 포함, AI 추천이 있으면 첫 번째 배치 + "(추천)" 표기
- 자유 입력은 항상 허용 — 선택지는 제안이지 강제가 아니다

### AI 응답
- 스트리밍 표시 (문서 생성 포함)
- 실패 시: 오류 메시지 + [다시 시도], 진행 중이던 답변은 로컬 보존 (입력 유실 금지)
- 생성 중 이탈 가능 — 백그라운드 완료 후 프로젝트 홈에 배지

### Decision Panel
- 결정 카드: 제목 + 요약 1줄, 클릭 시 전체 + 근거(원 질문·답변) + [수정] [무효화]
- `미정` 결정은 ⚠ 스타일로 상단 고정 — 단계 승인 전 리마인드

### 상태·피드백
- phase `stale` 배지: "상위 문서가 개정되었습니다 — 검토 필요" (차단하지 않음)
- 모든 저장은 자동 저장 + 명시적 버전 스냅샷은 [저장] 시점

## 6. Design System

- **컴포넌트**: shadcn/ui 기본. 커스텀은 QuestionCard, DecisionPanel, PipelineStepper, DocViewer 4종만
- **타이포**: 시스템 폰트 스택 + 문서 본문은 Pretendard, 코드·Markdown은 mono
- **컬러 토큰** (Tailwind CSS variables, 다크모드 기본 지원):
  - `--primary`: 파이프라인·CTA (인디고 계열)
  - `--phase-done / --phase-active / --phase-locked / --phase-stale`: 단계 상태 전용 토큰
  - `--decision / --decision-pending`: 결정 확정(녹색 계열) / 미정(앰버 계열)
- **레이아웃**: 데스크톱 3-pane(세션) / 2-pane(문서), 모바일 단일 컬럼 + 하단 시트
- **접근성**: 키보드로 질문 응답 가능 (선택지 숫자키, Enter 확정), 단계 상태는 색+아이콘 병행

## 7. Empty & Edge States

| 상황 | 처리 |
|---|---|
| 프로젝트 0개 | "첫 제품을 설계해 보세요" + 예시 아이디어 3개 칩 (클릭 시 입력창에 채움) |
| 세션 중 브라우저 종료 | F5 재개 플로우 — 입력 중 답변은 로컬 스토리지 보존 |
| AI 생성 실패 | 부분 생성물 보존 + 실패 지점부터 재시도 |
| 문서 승인 후 상위 단계 수정 | 하위 문서 `stale` 배지, 사용자가 [재생성] 또는 [무시] 선택 |
| 미정 결정이 남은 채 단계 승인 시도 | 경고 모달: 미정 목록 표시 → "그래도 승인" 허용 (차단 아님, No Guessing 기록 유지) |

## 8. MVP 경계

- **MVP**: F1~F5 플로우 전체, SC-05 3-pane, 문서 편집·버전, Handoff zip/복사
- **Should**: F6 회고, 부분 재생성, 프롬프트 템플릿 UI
- **Won't (MVP)**: 실시간 협업 커서, 문서 코멘트, Figma 연동, 온보딩 투어
